# Архитектура

## Контекст

Официальный Telegram-клиент для [MafiaTracker](../MS). Полностью отдельное
приложение: свой репозиторий, свой аккаунт PythonAnywhere, своя (минимальная)
БД либо вообще без БД. Взаимодействие с основным сайтом — **только через
его API**, никакого прямого доступа к его базе данных.

## Технологический выбор

Хостинг — PythonAnywhere, синхронная WSGI-модель (запрос-ответ, без вечно
работающего процесса). Приём сообщений от Telegram — **вебхук**, не
long-polling (long-polling требует постоянного цикла — Always-on Task,
недоступно/дорого на простом тарифе).

Библиотека — **`pyTelegramBotAPI` (`telebot`)**, синхронная. `aiogram` и
`python-telegram-bot` v20+ — async-first, заточены под собственный event
loop; встраивать их в синхронный Flask-запрос на каждый вебхук-вызов
возможно, но не по их естественной логике. `telebot` штатно поддерживает
паттерн "Flask-роут получает JSON → `bot.process_new_updates(...)`" и
держит тот же синхронный стиль, что и весь основной проект.

## Слои

```
Telegram/сайт → Flask-роут → handlers/ → services/ → api_client/ → основной сайт
                                  ↓
                              presenters/ (чистые функции, без сети)
```

- **handlers/** — тонкие: разобрать апдейт/callback, вызвать `services`,
  отдать результат `presenters`, отправить/отредактировать сообщение.
  Никакой бизнес-логики, никаких прямых HTTP-вызовов.
- **presenters/** — чистые функции `данные → (текст, клавиатура)`. Ноль
  I/O — тестируются без мока сети и без реального токена бота (см.
  `tests/test_presenters_start.py`).
- **services/** — бот-специфичные правила ("не привязан — предложи
  привязать"), склейка `handlers` ↔ `api_client`.
- **api_client/** — единственное место, знающее про HTTP к основному
  сайту. Типизированные исключения (`ApiTimeout`, `ApiUnauthorized`,
  `ApiNotFound`, `ApiServerError`) вместо голых `requests.exceptions` —
  хендлеры ловят их и отвечают дружелюбным сообщением, не трейсбеком.
- **webhooks/** — обработка входящих `POST /events/<type>` от основного
  сайта (fire-and-forget уведомления о событиях). Пока не реализовано —
  появится вместе с соответствующими фичами на стороне основного сайта.

## Привязка аккаунта — без своей таблицы

Привязка происходит **целиком на основном сайте** через официальный
Telegram Login Widget (пользователь жмёт кнопку в профиле на сайте,
подтверждает в Telegram — без захода в диалог с ботом заранее). Значит:

- Единственный источник правды о привязке — колонка `Player.telegram_id`
  на основном сайте.
- **У бота НЕТ собственной таблицы "telegram_id ↔ player_id".** При каждом
  входящем сообщении бот спрашивает у API `GET /api/v1/bot/resolve?
  telegram_id=...` — кто это. Меньше движущихся частей, невозможен
  рассинхрон двух копий одних и тех же данных.
- Локальная БД у бота появится только если что-то по-настоящему
  бот-специфичное потребует состояния между запросами, которое не влезает
  в `callback_data` (например, многошаговый диалог) — по факту
  необходимости, не заранее.

## Безопасность

Три разных направления доверия, не путать:

1. **Telegram → бот** — секретный сегмент пути (`/telegram/webhook/<64
   случайных символа>`) + `secret_token`, который Telegram сам присылает
   в заголовке `X-Telegram-Bot-Api-Secret-Token` на каждый вызов (задаётся
   при регистрации вебхука). Проверка — `bot/security.py::verify_telegram_secret`.
2. **Бот → основной сайт** — один серверный секрет
   (`MAIN_API_SERVICE_TOKEN`, `Authorization: Bearer`), доказывающий "это
   наш бот-сервер", плюс `telegram_id` как параметр запроса — сайт сам
   резолвит его в игрока. Секрет никогда не виден пользователю бота.
3. **Основной сайт → бот** (fire-and-forget уведомления о событиях) —
   общий HMAC-секрет (`INCOMING_EVENT_SECRET`), подпись тела запроса в
   заголовке `X-Signature`, сверяется константным сравнением. Проверка —
   `bot/security.py::verify_event_signature`.
4. **Telegram Login Widget на основном сайте** (это про сайт, не про
   бота, но упомянуто здесь для полноты картины) — верификация подписи
   виджета требует секрет `SHA256(TELEGRAM_BOT_TOKEN)`, то есть основному
   сайту тоже нужен `TELEGRAM_BOT_TOKEN` — требование протокола Telegram,
   не архитектурный выбор этого проекта.

## Обработка ошибок

`bot/__init__.py::create_app()` оборачивает `process_new_updates` в
try/except: ошибка внутри хендлера (в т.ч. реальный сбой Telegram API при
отправке) логируется, но HTTP-ответ Telegram'у всегда `200` — иначе
Telegram будет ретраить один и тот же апдейт бесконечно. Ошибки при
обращении к API основного сайта ловятся в хендлерах по типизированным
исключениям `api_client.exceptions` → дружелюбное сообщение пользователю.

## Конфигурация и секреты

`bot/config.py` — тот же стиль, что `app/config.py` основного проекта:
плоский `os.environ.get()`, обязательные переменные падают с понятной
`RuntimeError`, если не заданы — без тихого fallback. **Важно**: импорт
`config_map` происходит ВНУТРИ `create_app()`, не на уровне модуля —
иначе любой импорт `bot.*` (в т.ч. из тестов) требовал бы реальных
секретов просто из-за того, что Python исполняет `__init__.py` пакета при
импорте любого его подмодуля. Та же ловушка уже была учтена в
`app/__init__.py` основного проекта.

Переменные — см. [.env.example](.env.example).

## Кеширование

In-process TTL-кэш (появится вместе с первыми read-heavy эндпоинтами —
лидерборд, каталог магазина). Персональные данные (баланс, профиль) —
всегда без кэша. Redis/Memcached не нужны на этом масштабе.

## Логирование

Стандартный `logging`, структурированные записи. `bot/logging_config.py`.

## Масштабируемость

Вебхук-модель = приложение стейтлес между запросами (кроме опционального
in-process кэша) — масштабируется ровно как обычный Flask-сайт.

## Обновления/деплой

Свой git-репозиторий, свой аккаунт PythonAnywhere, `deploy.sh` (git pull +
touch WSGI-файла) — тот же проверенный паттерн, что у основного проекта.

## Зависимости от основного проекта

Функционал бота дальше этого инкремента упирается в изменения основного
сайта, которые ещё не сделаны:

| Нужно на основном сайте | Для какой фичи бота | Статус |
|---|---|---|
| `Player.telegram_id` + Login Widget | Привязка аккаунта, `/resolve` | ✅ Готово (MS commit `98e6db1`) |
| `Game.round_number` + `TournamentService.generate_next_round()` | Авторассадка следующего раунда | ✅ Готово (MS commit `ea0fe90`) |
| `BotNotifyService` + `INCOMING_EVENT_SECRET`/`BOT_EVENTS_URL` — событие `next-slot` | Уведомление "Твой слот — N" | ✅ Готово (MS commit `4d081fd`, бот `e57405b`) |
| `/api/v1/bot/resolve`, `/profile`, `/stats`, `/compare`, `/account/unlink` | Резолв привязки, профиль, статистика, сравнение, отвязка | ✅ Готово (MS commit `e6af578`) — бот пока их не вызывает, см. ниже |
| `/api/v1/bot/*` — рейтинг/история/экономика/достижения/титулы/турниры/fantasy | Остальные разделы меню | Не начато |
| Остальные fire-and-forget события (завершение игры, достижение/титул, перекуп Mythic/Ultra, fantasy, подарок, сезонная награда) | Остальные проактивные уведомления | Не начато — `next-slot` уже готовый шаблон для них |

Готовность `Player.telegram_id`/Login Widget и первой партии
`/api/v1/bot/*` означает: эндпоинты на MS реально работают и
протестированы (`app.test_client()`), но `bot/api_client/` их ЕЩЁ НЕ
вызывает — нет ни методов-обёрток под конкретные эндпоинты, ни хендлеров,
которые бы их использовали. `/start` пока отвечает статичным текстом.
Это и есть следующий инкремент.

Готовность авторассадки означает: `generate_next_round` считает и
возвращает (`ServiceResult.data`) всё нужное для уведомления —
`assignments` (player_id/game_id/table_number/seat_number/round_number на
каждого) и `resting_player_ids`.

Готовность `next-slot`-события означает: `finish_game` на MS уже реально
шлёт его боту сразу после `generate_next_round` (`app/services/
bot_notify_service.py::BotNotifyService.send_event`), а
`bot/webhooks/events.py::dispatch` уже реально его обрабатывает и шлёт
"Твой слот — N" через `TeleBot.send_message` — весь путь end-to-end
рабочий и проверен между двумя реально запущенными серверами. Остальные
типы событий (см. таблицу выше) используют тот же `BotNotifyService` +
тот же `EVENT_HANDLERS`-реестр — `next-slot` теперь готовый шаблон для
них, а не то, что нужно проектировать с нуля.

Каждый пункт реализуется отдельным согласованным инкрементом на стороне
основного репозитория (MS), не здесь.
